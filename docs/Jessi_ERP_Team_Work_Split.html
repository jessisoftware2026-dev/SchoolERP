Jessi ERP — Two-Team Work Split Plan

Module / domain ownership · AI-Ready Multi-Tenant School ERP · © 2026 Jessi Software

1. Approach

Work is split by module/domain: each team owns complete vertical slices — the Next.js UI, the NestJS API module, the data model, and the related AI endpoints for the features they own. This maximises parallel delivery and minimises hand-offs. Both teams build on a shared foundation (auth, RBAC, multi-tenancy, design system) that is delivered first and maintained jointly.

Reference pattern: the existing Students module (UI → NestJS → Prisma/Postgres, tenant-scoped) is the template every other module copies.

7. Cross-Team Interfaces & Rules

  • Shared Prisma schema: one schema file — coordinate migrations through the platform squad; never two people editing it at once. Each team adds its own models/relations via PR review.
  • Shared UI library: components live in apps/web/src/components; changes that affect both teams go through the platform squad.
  • API contracts: every module exposes REST under its own path, tenant-scoped via @CurrentSchool() + TenantGuard; document in Swagger.
  • Reports/MIS reads everything: Team B's reporting depends on all modules — agree on stable read models / views early.
  • Dependencies flow forward: Fees → Accounts; Timetable → HR (teachers); Attendance/Exams → AI. Build providers before consumers (see phase column).

8. Definition of Done (every module)

  • All data tenant-scoped by schoolId; queries filtered by tenant.
  • RBAC permissions added to the seed for each role; UI gated via usePermissions.
  • DTO validation + Swagger docs on every endpoint.
  • Follows the Students vertical-slice pattern (controller → service → Prisma).
  • Responsive UI, light/dark mode, matches the design system.
  • Unit tests for service logic; basic e2e for critical flows.
  • Seed/sample data so the module is demoable.
